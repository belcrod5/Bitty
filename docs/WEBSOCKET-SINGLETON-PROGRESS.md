## Confirmed Facts

- `/runner-ws` already carries multi-channel JSON traffic.
- The design requires one app-wide WebSocket for JSON control traffic only.
- Heavy binary or bulk transport remains HTTP.
- Start-style operations require stable logical ids before moving onto the singleton.
- Server connection state must stop treating one physical WebSocket as one feature operation.
- `expo/src/features/runnerWs/types.ts` has `requestId`, `sessionId`, `threadId`, and `streamId`, but not `operationId`.
- `private_runner/src/server-runtime.mjs` stores `llmRelay` and `attachedTtsJobId` in the `/runner-ws` connection handler.
- `expo/src/features/ws/webSocketAuth.ts` is only the low-level authenticated WebSocket factory.

## Recommended First PR

The user chose not to split into small PRs. Keep one branch, but still implement in safe sequential tasks.

Start with the server/client contract foundation:

- add `operationId` to the shared runner-ws envelope types and validators.
- add focused tests for envelope parsing/validation.
- prepare server-side operation identity handling behind existing `/runner-ws` behavior.

This keeps the first PR small and avoids moving feature traffic before the server can route logical operations safely.

## Sequential Task Order

1. Add `operationId` to the runner-ws envelope contract on client and server, plus focused adapter/parser tests.
2. Make server `/runner-ws` route logical LLM/TTS ownership by `sessionId`/`threadId`/`operationId`/`streamId`, not only by the physical WebSocket.
3. Add `RunnerWebSocketManager` and context with `control:ready` handshake, snapshots, heartbeat, reconnect, AppState lifecycle, pending request cleanup, and subscriber routing.
4. Move diagnostics/status ping to the manager.
5. Move relay observer traffic to manager subscriptions/sends.
6. Move LLM turn and one-shot JSON-RPC traffic, including WebSocket-local JSON-RPC id mapping.
7. Move stream TTS control traffic and replace socket-presence liveness with job/stream state.
8. Run focused tests during each phase and broad validation at the end.

3. Low-risk diagnostics/status traffic migration.
